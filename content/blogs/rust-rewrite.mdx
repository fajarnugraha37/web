# Why I Rewrote It In Rust

Let me save you the Reddit argument: this isn't about Rust being morally superior to C++. It's about a specific class of performance-critical services where the GC pause budget is zero, the memory budget is fixed, and the oncall rotation is tired of being paged at 3am because the heap exploded.

Rust was the answer. It might not be yours. Let's figure that out before you rewrite everything and alienate your entire team.

## 1. When the Rewrite Is Justified

You have a valid case for Rust when:

- Latency spikes correlate directly with GC runs in your APM traces
- You're writing a hot path that runs millions of iterations per second
- You genuinely cannot afford an allocator on the critical loop
- You've already profiled, and the bottleneck is memory allocation, not logic

If your bottleneck is a poorly written SQL query or a missing index, rewriting in Rust will produce a fast, poorly-designed data pipeline on top of the same slow query. Congratulations.

## 2. The Actual Cost of Zero-Copy

The Rust pitch: zero-copy parsing, stack allocation, no GC jitter. The reality: you will spend two weeks fighting the borrow checker before you write a single line of business logic.

```rust
fn parse_frame<'a>(buf: &'a [u8]) -> Result<Frame<'a>, ParseError> {
    let header = buf.get(..4).ok_or(ParseError::TooShort)?;
    let length = u32::from_be_bytes(header.try_into().unwrap()) as usize;
    let payload = buf.get(4..4 + length).ok_or(ParseError::Incomplete)?;
    Ok(Frame { header, payload })
}
```

That lifetime annotation `'a` is doing real work. The payload borrows from the original buffer, zero copies. The compiler guarantees it. The tradeoff is that you must think about ownership explicitly, which Java and Go quietly let you ignore in exchange for a pause at inopportune moments.

## 3. Interop: The Part Nobody Talks About

If you can't rewrite everything at once (and you can't), you'll use FFI. Rust's FFI with C is well-documented and not painful. Rust's FFI with Java via JNI is a different kind of suffering.

```rust
#[no_mangle]
pub extern "C" fn process_batch(ptr: *const u8, len: usize) -> i64 {
    // SAFETY: caller guarantees ptr is valid for `len` bytes
    let data = unsafe { std::slice::from_raw_parts(ptr, len) };
    compute(data) as i64
}
```

This is fine for offloading one hot function. Don't build your entire service on JNI calls. It will look like someone tried to marry two languages and neither family approved.

## Conclusion

Rewrite in Rust when the profiler tells you to, not when Hacker News tells you to. Profile first. Identify the hot path. Prove the bottleneck is memory and allocation, not logic. Then rewrite that specific component. Not the whole system. Not yet.

The borrow checker has seen your code. It has concerns.
